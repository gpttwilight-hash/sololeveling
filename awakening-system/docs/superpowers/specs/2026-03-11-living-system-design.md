# Дизайн: «Живая Система» — Daily Pull Mechanics

**Дата:** 2026-03-11
**Цель:** Создать механики, которые дают пользователю причину открывать приложение каждый день без внешних напоминаний.

---

## Проблема

Приложение отлично работает когда открыто, но не создаёт pull-механики для ежедневного открытия. Пользователи знают свои задачи и перестают заходить через 3-5 дней — "я и так помню что делать".

---

## Решение: Три независимых механики

### 1. Daily Dispatch — «Система обращается к тебе»

Каждую полночь для каждого пользователя выбирается персональный Dispatch — нарративное сообщение + специальный квест дня с повышенной наградой.

**Алгоритм выбора шаблона:**
```
selectDispatch(profile, date):
  attributeOfDay = CYCLE[dayOfYear % 5]  // STR→INT→CHA→DIS→WLT
  context = {
    rankTier: E/D/C/B/A/S,
    streakBand: 0 | 1-2 | 3-6 | 7-13 | 14-29 | 30+,
    weakestAttribute,
    dayOfWeek
  }
  → детерминированный выбор из 60 шаблонов
```

**60 шаблонов** покрывают матрицу: 6 рангов × 5 контекстов стрика × корректировка по атрибуту.

**Данные:**
```sql
daily_dispatch (
  id, user_id, date,
  template_id,        -- ссылка на шаблон
  narrative_text,     -- текст от Системы
  bonus_quest_title,  -- название задания дня
  bonus_quest_xp,     -- +50% к базовому XP атрибута дня
  bonus_quest_coins,
  attribute_focus,    -- атрибут дня
  is_completed,
  expires_at          -- полночь текущего дня
)
```

**UI:** Блок в верхней части дашборда (после HeroQuest), с таймером обратного отсчёта до сброса.

---

### 2. Portal of the Day — «Данжен на 24 часа»

Ежедневный глобальный портал из 3 последовательных шагов. Шаги разблокируются по одному — повод возвращаться в течение дня. Закрывается в полночь.

**Структура:**
- Шаг 1: простой (5-10 мин) — низкий порог входа
- Шаг 2: средний (15-20 мин)
- Шаг 3: финальный — разблокируется только после шага 2

**Данные:**
```sql
portal_templates (
  id, title, description,
  attribute,           -- атрибут портала
  steps jsonb,         -- [{title, description, xp_reward}]
  total_xp_reward,
  badge_name           -- уникальный бейдж за полное завершение
)

daily_portal_log (
  date PRIMARY KEY,
  template_id          -- какой портал активен сегодня
)

user_portal_progress (
  user_id, date,
  steps_completed int, -- 0, 1, 2, 3
  is_finished bool,
  started_at, finished_at
)
```

**30 шаблонов** порталов, ротация по атрибуту дня + ручной выбор.

---

### 3. Живой профиль — «Охотник эволюционирует»

Профиль визуально меняется в зависимости от активности. При открытии приложения пользователь видит изменения.

**Аура охотника** (цвет + интенсивность по стрику):
```
0-2 дня   → серая тусклая
3-6 дней  → мерцающая, цвет топ-атрибута
7-13 дней → пульсирующая яркая
14+ дней  → многослойная (два доминирующих атрибута)
```

**Ежедневная фраза Системы** в ProfileCard — выбирается из пула по контексту (стрик, топ-атрибут, уровень).

**Milestone-события** на дни 7, 14, 30, 100 — fullscreen overlay при открытии приложения с уникальным титулом:
- День 7 → "Пробуждённый"
- День 14 → "Закалённый"
- День 30 → "Несломленный"
- День 100 → "Легенда Системы"

10 заготовленных milestone-текстов с подстановкой имени охотника.

**Изменения в profiles:**
```sql
ALTER TABLE profiles ADD COLUMN last_milestone_shown int DEFAULT 0;
-- aura_level вычисляется из current_streak, не хранится
```

## Технический стек

- **Шаблоны:** TypeScript-константы в `src/lib/game/dispatch-templates.ts` и `portal-templates.ts`
- **Выбор шаблона:** детерминированные функции, без внешних API
- **Cron:** Supabase Edge Function или Next.js route handler на `/api/daily-reset`
- **UI:** новые клиентские компоненты в `src/components/dashboard/`
- **БД:** 3 новые таблицы + 1 ALTER на profiles

---

## Что НЕ входит в скоуп

- Gemini/Claude API интеграция (заменена шаблонами)
- Социальные функции (гильдии, рейтинги)
- Изменения существующих механик (квесты, XP, ранги)

---

## Критерии успеха

- D1 открытий: пользователь возвращается на следующий день
- Портал завершён: >40% пользователей проходят все 3 шага
- Dispatch прочитан: >80% пользователей взаимодействуют с блоком
